## 1. 项目概况与目标

### 1.1 项目定位

**SortJudge（SortJudge）** 是一个面向算法教学的 Web 评测平台，让学生通过浏览器内置 IDE 编写排序算法代码，提交后由系统自动评测、识别算法类型、给出智能反馈、评分，并支持排序过程的可视化回放。

### 1.2 核心目标

| 目标 | 描述 |
|------|------|
| **自动评测** | 沙箱执行学生代码，逐用例比对输出，自动判断 pass/fail/error/timeout |
| **算法识别** | 基于 AST（抽象语法树）静态分析，识别学生使用了哪种排序算法（冒泡/选择/插入/快速/归并/堆） |
| **智能反馈** | 纯规则驱动（不依赖 LLM），通过 traceback 模式匹配 + AST 静态检查给出针对性教学提示 |
| **多规则判定** | 支持 strict（严格排序）、stalin（子序列）、stable（稳定排序）、topk（前 K 有序）四种排序规则 |
| **可视化回放** | 收集排序步骤快照，通过 Canvas 动画回放排序过程，支持变速/单步/暂停/音效 |
| **复杂度分析** | 多规模（n=100/500/1000/2000）实测反推时间复杂度 O(n) / O(n log n) / O(n²) |
| **代码查重** | 教师可基于 AST 余弦相似度检测学生之间代码雷同 |
| **10 分制评分** | 四维度评分：正确性(0-4)、算法匹配度(0-2)、代码质量(0-2)、效率(0-2) |

### 1.3 关键设计原则

- **核心引擎完全解耦**：`app/core/` 下所有模块纯 Python 函数，零 Flask 依赖，可独立测试
- **可视化冗余设计**：步骤采集始终执行（无论评测结果），前后端分离，数据先行
- **零外部 API 依赖**：反馈引擎纯规则驱动，不依赖任何 LLM 或外部服务
- **安全性**：沙箱子进程降权执行 + timeout + AST 静态预检查危险函数调用
- **中英双语**：前端界面同时支持中文和英文

## 5. 核心引擎详解

### 5.1 模块职责总览

| 模块 | 文件 | 核心能力 | 输入 | 输出 |
|------|------|----------|------|------|
| 沙箱执行器 | `sandbox.py` | 隔离执行学生代码 | code + stdin | RunResult {stdout, stderr, returncode, timed_out} |
| 算法识别器 | `recognizer.py` | AST 静态识别算法类型 | Python 源代码 | "bubble", "quick", "merge", ... |
| 排序判定器 | `validator.py` | 按规则判定排序正确性 | output + input + rule | {passed, reason} |
| 输出比对器 | `comparator.py` | 比对输出与期望 | actual + expected + rule | {passed, reason} |
| 反馈引擎 | `feedback.py` | 诊断错误并给出教学提示 | code + run_result | {type, hint, details} |
| 步骤采集器 | `step_collector.py` | 采集排序步骤快照 | code + input_arr | list[{seq, array_state, op, i, j}] |
| 评分引擎 | `grader.py` | 10 分制四维度评分 | code + 评测结果 | {total, correctness, algorithm, quality, efficiency} |
| 复杂度估算 | `complexity.py` | 多规模实测反推复杂度 | code + sizes | {estimated, timings, ratios, max_ratio} |
| 相似度检测 | `similarity.py` | AST token 余弦相似度 | code_a + code_b | float(0.0~1.0) |
| 算法档案 | `algo_profiles.py` | 6 种算法的理论复杂度 | algo_type | {best, avg, worst, space, stable} |
| 参考示例 | `examples.py` | 9 种算法的标准实现 | 算法 key | 示例代码字符串 |

### 5.2 算法识别逻辑 (recognizer.py)

基于 Python `ast` 模块的静态分析，通过检查 AST 特征判定算法类型：

```
特征检测顺序（从特殊到一般）：
  1. 递归调用? + list.pop(0)?        → merge (归并排序)
  2. 递归调用? + 无 pop(0)?          → quick (快速排序)
  3. 递归调用? + heapq 相关?         → heap (堆排序)
  4. 嵌套循环 + 相邻交换?            → bubble (冒泡排序)
  5. 嵌套循环 + min() 查找?          → select (选择排序)
  6. 嵌套循环 + while 内循环+移位?   → insert (插入排序)
  7. 不匹配任何已知模式              → unknown (未知)
```

### 5.3 排序规则判定 (validator.py)

| 规则 | 判定条件 | 典型题目 |
|------|----------|----------|
| `strict` | 输出 = 输入排序后的排列（长度相同 + 单调 + 排列） | 冒泡/快速/归并/选择/插入 |
| `stalin` | 输出是输入的子序列 + 输出单调递增 | 斯大林排序 |
| `stable` | strict + 相等元素的相对顺序不变 | （保留扩展） |
| `topk` | 前 K 个元素排序正确 | （保留扩展） |

附加检测：**off-by-one 错误**（输出长度差 1）、部分有序但不完全正确等。

### 5.4 智能反馈引擎 (feedback.py)

**双层反馈架构**：

**第一层：Traceback 模式匹配**（8 种常见错误）
- `NameError` → 变量未定义提示
- `TypeError` → 类型错误（如对 int 调用 len）
- `IndexError` → 数组越界（检查循环边界或空数组处理）
- `RecursionError` → 递归深度超限（检查基准条件）
- `AttributeError` → 属性错误（如 list.sort）
- `KeyError` → 字典键错误
- `ValueError` → 值错误
- 其他 → 通用运行时错误提示

**第二层：AST 静态检查**
- 危险函数调用检测（eval, exec, open, __import__, compile, file, input）
- sort 函数定义缺失
- sort 函数缺少 return 语句
- sort 函数参数列表异常

### 5.5 步骤采集器 (step_collector.py)

**三种策略**：

1. **TrackedList 策略**（当前使用）：
   - 使用 `TrackedList`（list 子类）拦截 `__setitem__`、`pop`、`append` 等操作
   - 自动记录每次数组变更的快照
   - 优点：零侵入，学生代码无需修改

2. **AST 插桩策略**（预留）：
   - 在 AST 层面自动插入记录代码

3. **Replay 策略**（预留）：
   - 基于已知算法类型推演排序步骤

### 5.6 评分引擎 (grader.py)

| 维度 | 分值 | 评分依据 |
|------|------|----------|
| **正确性** | 0-4 | 通过率: 100%→4, ≥75%→3, ≥50%→2, >0%→1, 0%→0 |
| **算法匹配** | 0-2 | 已知且匹配→2, 已知不匹配→1, 未知→0 |
| **代码质量** | 0-2 | 无安全风险+无结构问题→2, 有结构问题→1, 有安全风险→0 |
| **效率** | 0-2 | 实测≤门槛→2, 实测>门槛→1, 未分析但满分→1, 未分析→0 |

| **user** | 用户（学生+教师） | id | — | username UNIQUE, role |
| **problem** | 题目 | id | created_by → user.id | algo_type, sort_rule, is_meme |
| **test_case** | 测试用例 | id | problem_id → problem.id | input_data (JSON), is_public |
| **submission** | 提交记录 | id | user_id, problem_id | status, score, recognized_algo |
| **submission_detail** | 评测详情（每用例一条） | id | submission_id, test_case_id | passed, feedback |
| **sort_step** | 排序步骤快照 | id | detail_id → detail.id | seq, array_state (JSON), op |
| **complexity_analysis** | 复杂度分析 | id | submission_id (UNIQUE) | estimated, timings (JSON), ratio |

### 6.2 实体关系（E-R）

```
user   ──1───0..*── submission       (一个用户可多次提交)
problem──1───0..*── test_case         (一道题有多个测试用例)
problem──1───0..*── submission        (一道题可被多次提交)
submission──1───0..*── detail         (一次提交评测多组用例)
detail  ──1───0..*── sort_step        (一组用例有多步排序快照)
submission──1───0..1── complexity     (一次提交最多一次分析)
```

### 6.3 status 状态机

```
                   ┌────────────────┐
                   │    pending     │ (初始状态)
                   └───┬───┬───┬────┘
         全部通过      │   │   │  超时
           ▼          │   │   └──────▶  timeout
         pass         │   │ 运行错误
                      │   └──────────▶  error
                 部分通过
                      ▼
                    fail
```

## 7. 运行流程与数据流

### 7.1 完整评测管线

```
学生编写代码 → POST /submit/{pid}
 │
 ├─ 1. 空代码检查 → 400
 ├─ 2. AST 静态检查（安全预检）→ 含危险函数则 400
 ├─ 3. 算法识别 (recognizer.py) → "bubble(冒泡排序)"
 ├─ 4. 创建 submission (status=pending, flush)
 │
 ├─ 5. 逐用例循环 ──────────────────────────────────┐
 │    ├─ 包装代码 (添加 stdin/stdout 样板)            │
 │    ├─ 沙箱执行 (subprocess, timeout=5s)            │
 │    │   ├─ 超时 → detail(feedback=超时提示), continue│
 │    │   ├─ 运行错误 → diagnose(feedback) → detail    │
 │    │   └─ 成功 → 比对输出 (comparator.py)           │
 │    │       ├─ 通过 → detail(passed=true)            │
 │    │       └─ 不通过 → detail(feedback=原因)        │
 │    ├─ 步骤采集 (step_collector → TrackedList)      │
 │    └─ 写入 sort_step 记录                          │
 └──────────────────────────────────────────────────┘
 │
 ├─ 6. 汇总 status + score, commit
 ├─ 7. 评分 (grader.py) → 10分制报告
 └─ 8. 返回 JSON 结果给前端
```

### 7.2 按需触发（异步）

- **复杂度分析**：`POST /submit/api/complexity/{sid}` → 多规模实测 → 写入 complexity_analysis
- **可视化回放**：`GET /submit/api/steps/{did}` → 读取 sort_step → Canvas 动画
- **代码查重**：`POST /teacher/similarity/{pid}` → AST 余弦相似度 → 相似度对列表

### 9.2 可视化组件详解

**Canvas 动画特性**：
- 柱状图展示数组（高度 = 值，颜色区分操作元素）
- 操作类型色彩编码：compare(黄)、swap(红渐变)、set(绿)、done(蓝)
- 播放控制：▶ 播放 / ⏸ 暂停 / ⏭ 单步 / 重置
- 速度控制滑块（1x~10x）
- 进度条 + 步数显示
- **Web Audio API 音效**：
  - 比较操作：短促 "嘀" 声（~600Hz, 0.05s）
  - 交换操作：稍高 "叮" 声（~880Hz, 0.08s）
  - 完成：大三和弦（C-E-G, 0.3s）
- 代码行高亮同步（对应 CodeMirror 行号）
- 步骤解释面板（op + i + j + note）

## 13. 亮点与改进建议

### 13.1 项目亮点

1. ⭐ **核心引擎完全解耦**：`core/` 模块零 Flask 依赖，纯 Python 函数，可独立测试和复用
2. ⭐ **可视化冗余设计**：步骤采集始终执行，与渲染分离，数据先行
3. ⭐ **纯规则反馈**：不依赖 LLM API，反馈即时、零成本、可预测
4. ⭐ **安全性分层**：AST 预检 + subprocess 隔离 + timeout + 降权，多层防护
5. ⭐ **整活专区**：Stalin/Bogo/Sleep/MercifulStalin 四种创意排序，寓教于乐
6. ⭐ **中英双语**：UI 原生支持语言切换
7. ⭐ **评测管线健壮**：try/rollback 保证异常时数据一致性
8. ⭐ **测试覆盖全面**：~75+ 用例，核心引擎 100% 单元测试覆盖
9. ⭐ **一键启动**：AAA启动器.bat 实现零门槛部署
10. ⭐ **Canvas 动画音效**：全屏可视化回放 + Web Audio API + 代码行同步高亮

### 13.2 改进建议

| 类别 | 建议 | 优先级 |
|------|------|--------|
| **功能** | 添加 Heap Sort 题目（目前引擎支持但 seed 无单独堆排序题） | 中 |
| **功能** | 实现 topk 和 stable 规则的实际题目 | 低 |
| **功能** | 添加排行榜/积分系统 | 低 |
| **安全性** | 添加提交频率限制（rate limiting） | 高 |
| **安全性** | 沙箱使用 Docker 容器隔离（当前仅 subprocess） | 中 |
| **UX** | 编辑器支持自动保存草稿（localStorage） | 中 |
| **UX** | 可视化增加更多算法相关的 UI 指示器 | 中 |
| **性能** | 复杂度分析结果缓存（避免重复计算） | 中 |
| **代码质量** | `step_collector.py` 的 AST 插桩和 Replay 策略待完善 | 低 |
| **部署** | 生产环境升级：Gunicorn + PostgreSQL + Docker | 中 |
| **文档** | README.md 内容待补全 | 高 |
| **测试** | 增加前端 E2E 测试（Selenium/Playwright） | 低 |
